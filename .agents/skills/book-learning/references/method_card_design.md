# Method Card Design

A method card turns book knowledge into an actionable reasoning tool.

A method card is valid only if it can help the Agent perform a real task.

Good method cards:

- evaluate plans
- check arguments
- structure writing
- make decisions
- identify hidden assumptions
- analyze risks
- guide retrospectives
- improve communication

Bad method cards:

- only summarize a chapter
- only define a concept
- only quote the book
- cannot be used in a user task
- have no output template
- have no applicable scene

Method cards should be extracted from:

- repeated frameworks
- explicit methods
- decision models
- diagnostic questions
- reasoning patterns
- evaluation checklists
- communication structures

## Extraction Rules

- Extract method cards only after `outputs/reading_notes.md` is complete and audited.
- Each book should produce at most 3-5 method cards.
- Prefer fewer, stronger cards over many shallow cards.
- A method card must include applicable scenes, non-applicable scenes, steps, output template, challenge questions, and next actions.
- A method card is not a summary. It is a callable thinking tool.
